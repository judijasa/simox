[.] One-to-many and/or many-to-one:
    There are job offers with a single OPEC but with many vacantes, each
    with a different muncipio and departamento.  Currently, we have a
    municipio column that is of string type.  Multiple municipios are simply
    separated by a comma. The departamento is being encoded by a single id,
    which it doesn't give an opportunity to encode the multiple departamento ids
    of the multiple vacantes. Solve this problem. Possible solutions:
    1.  Use ordered array int types to store the id of  municipios (challenging
        for non-api approaches), id of departamentos and number of vacantes per
        location, for each OPEC number. Do this for the job_offer_snapshot table.
    2.  Use a table where OPEC is not UNIQUE, to store the locations (municipio
        and departamento) and number of vacantes per location.

    Both solutions can be used. Solution 1 is good because it is the correct
    way to keep all info about a given job offer in the snapshot table which
    is meant to be a historical record of the simo website/api. It has the
    drawback that array types are not good to be used to search for rows in a db.
    This is solved by solution 2, which stores the same data but in multiple rows
    rather than a single array.
